# magnet-waitlist-api

**A tiny, single-purpose waitlist signup API for [gomagnet.ai](https://gomagnet.ai)** — collect emails, dedupe them, and read them back with an admin token.

An Express 5 + node-postgres service that backs the waitlist form on **gomagnet.ai**. It does one job: accept a POST from the marketing site, validate and normalize the email, store it in Postgres (deduped on email), and expose a token-gated read endpoint so you can pull the list. It ships with a `render.yaml` blueprint that stands up the web service and its database together.

Two deliberate touches keep it safe to run in the open: Render PR previews auto-switch to an **in-memory store** (`IS_PULL_REQUEST=true`) so preview deploys never touch production data, and CORS is hand-rolled to allow `gomagnet.ai`, `*.onrender.com`, and localhost while rejecting everything else.

## Endpoints

| Method | Path                  | Purpose                                            | Auth                 |
|--------|-----------------------|----------------------------------------------------|----------------------|
| GET    | `/api/health`         | Health check (DB ping)                             | none                 |
| POST   | `/api/waitlist`       | Add a signup. Body: `{ "email": "...", "name": "...", "source": "..." }` (name + source optional) | none, rate-limited |
| GET    | `/api/waitlist`       | List signups (latest 1000)                         | `Bearer ADMIN_TOKEN` |
| GET    | `/api/waitlist/count` | Public total count                                 | none                 |

`POST /api/waitlist` is rate-limited to 5 requests per IP per minute. Emails are trimmed, lowercased, and validated with Zod before they hit the database.

## Local dev

```bash
npm install
DATABASE_URL=postgres://... ADMIN_TOKEN=test PORT=3001 npm run dev
```

With no `DATABASE_URL` set, the service falls back to the in-memory store, so you can boot and hit the endpoints without a database.

## Deploy

`render.yaml` provisions:
- Web service `magnet-waitlist-api` (autoDeploy on `main`, PR previews enabled)
- Postgres db `magnet-waitlist-db` (basic-256mb, PG16)
- `ADMIN_TOKEN` is auto-generated by Render at first deploy
- `ALLOW_ORIGINS` defaults to `https://gomagnet.ai,https://www.gomagnet.ai` (plus any `*.onrender.com`)

Connect this repo as a Render Blueprint and both services come up together.

## Schema

```sql
CREATE TABLE waitlist (
  id          SERIAL PRIMARY KEY,
  email       TEXT NOT NULL UNIQUE,
  name        TEXT,
  source      TEXT,
  ip          TEXT,
  user_agent  TEXT,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

Auto-migrated on boot. An idempotent `ALTER TABLE … ADD COLUMN IF NOT EXISTS name` runs against pre-existing tables, and inserts use `ON CONFLICT (email)` so re-signups are a no-op rather than an error.

## Reading signups

```bash
curl -H "Authorization: Bearer $ADMIN_TOKEN" https://magnet-waitlist-api.onrender.com/api/waitlist
```

The list endpoint omits IP and user-agent from its response; those columns are stored for abuse triage only.

## Tech stack

Express 5 · node-postgres (`pg`) · Zod · Node ≥ 20 (ESM) · deployed on Render.
